# 헤르메스 엔지니어링 설계 문서

> 하네스 엔지니어링 위에 헤르메스 러닝 루프를 접목하는 설계안.
> 구현 전 아이디어 정리 단계.

## 1. 개념 요약

### 하네스 vs 헤르메스

| 구분 | 하네스 (Harness) | 헤르메스 (Hermes) |
|---|---|---|
| 핵심 개념 | 통제와 제약 (Guardrails) | 학습과 진화 (Self-Evolving) |
| 규칙 생성 | 사람이 직접 작성 | AI가 대화에서 자동 추출 |
| 스킬 생성 | 사람이 큐레이션 | 반복 패턴 감지 후 자동 결정화 |
| 비유 | 잘 훈련된 직원의 매뉴얼 | 일할수록 노하우가 쌓이는 숙련공 |

### 둘의 관계

하네스와 헤르메스는 경쟁자가 아니다.

- **하네스** = AI가 움직일 수 있는 안전한 트랙
- **헤르메스** = 그 트랙 위를 달리며 가장 빠른 법을 기록하는 것

`ai-dev-setting`에 이미 하네스가 완성되어 있으므로, 헤르메스는 그 위에 **경량 러닝 루프**를 올리는 방식으로 구현한다.

## 2. 결정화 (Crystallization)

### 개념

> 대화 속 흘러가는 지식(액체)을 재사용 가능한 스킬 파일(결정체)로 굳히는 과정.

**예시:**

1회차: "pnpm 버전 오류났어. 8.x로 고정해줘" → 고침 → 휘발
2회차: "pnpm 오류 또 났네. 8로 맞춰줘" → 고침 → 휘발
3회차: 동일 패턴 감지 → **결정화 트리거**

```markdown
# pnpm-version-lock.md (자동 생성)
이 프로젝트는 pnpm 8.x를 사용한다.
package.json에 engines 필드로 고정할 것.
```

### 결정화 조건

- 동일 키워드/패턴이 **3회 이상** 반복될 때 트리거
- 횟수는 추후 프로젝트별로 조정 가능

## 3. 스킬 검색 + 주입 시스템

### 검색 풀 (전체 통합)

헤르메스는 스킬이 어디서 만들어졌는지 구분하지 않는다.
하네스가 만든 스킬과 헤르메스가 자동 생성한 스킬을 **동일한 풀에서 검색**한다.

```
[검색 풀]
├── assets/skills/harness-boundary-check/     ← 하네스가 만든 스킬
├── assets/skills/harness-reasoning-sandwich/ ← 하네스가 만든 스킬
├── .hermes/skills/pnpm-version-lock.md       ← 헤르메스 자동 생성
└── .hermes/skills/fastapi-error-pattern.md   ← 헤르메스 자동 생성
```

### 주입 방식

사용자 입력 발생 시 PreToolUse Hook이 키워드 추출 → DB(FTS5) 검색 → 관련 스킬을 프롬프트에 자동 삽입.

```
사용자: "pnpm 업그레이드해줘"
  ↓
PreToolUse Hook
  ↓
DB 검색: "pnpm" 키워드
  ↓
pnpm-version-lock.md 발견
  ↓
프롬프트 앞에 자동 삽입:
"⚠️ 관련 규칙: pnpm은 8.x로 고정할 것 (pnpm-version-lock.md)"
```

**효과**: 하네스로 만든 기존 스킬들도 더 잘 활용된다.

## 4. 기억 구조 (SQLite)

### 저장소 분리

| 저장소 | 경로 | 내용 |
|---|---|---|
| 전역 | `~/.hermes/global.db` | 전체 프로젝트 공통 패턴, 사용자 성향 |
| 프로젝트 | `[project]/.hermes/state.db` | 해당 프로젝트 전용 기억 |

### 핵심 테이블 (SQLite)

**session_history** (FTS5 전문 검색)
- `content` — 대화 내용 (검색 대상)
- `role` — user / agent / tool
- `timestamp`
- `project_id`

**harness_rules** (결정화된 규칙)
- `trigger_keywords` — 발동 키워드
- `instruction` — 실제 규칙 내용
- `source_session_id` — 근거 대화 ID
- `version` — 피드백으로 수정될 때마다 증가

**skill_index** (생성된 스킬 파일 메타데이터)
- `skill_path` — .md 파일 경로
- `keywords` — 검색용 키워드
- `created_at`, `used_count`

## 5. 알림 UX 설계

### 결정화 순간 즉시 알림 (박스 UI)

스킬 파일이 생성되는 순간 대화 중에 출력:

```
╔══════════════════════════════════╗
║  🧠 헤르메스 결정화 완료          ║
║  pnpm-version-lock.md  ← 신규생성 ║
╚══════════════════════════════════╝
```

- 대화 흐름을 끊지 않음
- 시각적으로 눈에 띔
- 사용자가 블랙박스 느낌 없이 신뢰 유지

### `/hermes-status` 슬래시 커맨드

언제든 전체 현황 확인:

```
/hermes-status
→ 전체 스킬 목록 + 최근 업데이트 날짜 + 활용 횟수
```

## 8. 자가 진화 스킬 — 로컬 vs 공통 전파

### 진화의 두 가지 방향

스킬이 진화할 때 **누구에게 적용되는가**에 따라 두 가지로 나뉜다.

**① 프로젝트 로컬 진화**
- 해당 프로젝트에만 해당하는 피드백
- `[project]/.hermes/skills/` 에만 저장
- `ai-dev-setting` 원본을 건드리지 않음
- 자동으로 처리

**② 공통 스킬 진화 (ai-dev-setting 반영)**
- 모든 프로젝트에 적용되어야 할 변경
- `ai-dev-setting/assets/skills/` 원본을 수정해야 함
- `update-all.sh` 로 전체 프로젝트에 전파 가능
- **반드시 사용자 승인 후 진행**

### 공통 진화 플로우

```
프로젝트A에서 공통스킬 진화 감지
         ↓
"이 변경을 ai-dev-setting 공통에 반영할까요?"
         ↓
사용자: "응"
         ↓
ai-dev-setting 브랜치 생성 + 스킬 파일 수정
         ↓
PR 생성 (사용자가 직접 머지)
         ↓
머지 후 update-all.sh 로 전체 프로젝트에 전파
```

### 설계 원칙

- **PR까지만 자동, 머지는 사람이** — 공통 스킬은 전체 프로젝트에 영향을 주므로 머지는 반드시 사용자 판단
- **로컬 진화와 공통 진화는 독립** — 사용자가 "아니오" 해도 프로젝트 로컬에서는 이미 진화된 상태 유지
- **공통 반영은 명시적 승인** — AI가 자동으로 공통 스킬을 건드리는 것은 금지

## 9. 3층 기억 구조

### 개념

Claude Code는 기본적으로 세션이 끝나면 모든 것을 잊는다.
3층 기억 구조는 이 망각 문제를 기억의 **종류별로 분리**해서 해결한다.

### 3개 레이어

**1층 — 세션 기억 (단기)**
- "방금 무슨 일이 있었나?"
- 현재 대화 내용, 도구 실행 결과
- 세션 종료 시 SQLite에 저장, FTS5로 검색 가능

**2층 — 영구 기억 (장기)**
- "이 사용자는 누구고 이 프로젝트는 어떤 환경인가?"
- 사용자 선호도, 프로젝트 환경 설정, 고정된 규칙
- 세션이 바뀌어도 유지
- 예: "이 프로젝트는 pnpm 9.x 사용", "이 유저는 간결한 코드 선호"

**3층 — 스킬 기억 (절차)**
- "이 일을 어떻게 하는가?"
- 결정화된 스킬 파일들 (.md)
- 자가 진화 스킬(§8)이 여기에 쌓임
- 실질적인 하네스 역할 담당

### 왜 3층으로 나누나?

한 군데 다 섞으면 검색 시 노이즈가 너무 많다.

```
"pnpm" 검색했을 때
❌ 한 군데: 6개월치 대화가 다 나옴
✅ 3층 분리: 스킬 기억에서 pnpm-version-lock.md 바로 나옴
```

### 기존 ai-dev-setting과의 매핑

| 기존 자산 | 3층 기억의 어디 | 방식 |
|---|---|---|
| `CLAUDE.md` | 2층 영구 기억 | 수동 |
| `assets/skills/` | 3층 스킬 기억 | 수동 |
| (없음) | 1층 세션 기억 | **신규 추가** |

1층 세션 기억이 새로 추가되는 핵심 부분이다.

## 10. 러닝 루프 (Retrospective)

### 개념

> 세션이 끝날 때마다 AI가 스스로 "이번에 뭘 배웠지?"를 정리하는 과정.
> 헤르메스의 심장 — 다른 모든 기능(결정화, 자가 진화, 기억 저장, 알림)이 여기서 연결된다.

### 5단계 흐름

```
1. 대화 종료
         ↓
2. 회고 시작
   "이번 세션에서 반복된 패턴이 있나?"
   "사용자가 수정 요청한 게 있나?"
         ↓
3. 판단
   일회성 → 1층 세션 기억에만 저장
   반복 패턴 → 결정화 트리거 (§2)
   스킬 수정 요청 → 자가 진화 트리거 (§8)
   공통 스킬 변경 → 공통 반영 여부 질문 (§8)
         ↓
4. 결정화 or 자가 진화 실행
         ↓
5. 알림 (§5)
   박스 UI로 사용자에게 보고
```

### 판단 기준표

| 판단 항목 | 결과 |
|---|---|
| 동일 패턴 3회 이상 | 결정화 트리거 |
| 사용자가 스킬 수정 요청 | 자가 진화 트리거 |
| 공통 스킬 변경 감지 | 공통 반영 여부 질문 |
| 특이사항 없음 | 1층 기억에만 저장 후 종료 |

### ai-dev-setting 진입점

기존 Stop Hook을 **확장**하는 방식으로 구현. 새 Hook을 만들지 않는다.

```
기존 Stop Hook: 피로도 방지 메시지 출력
확장 후:        피로도 방지 + 회고 로직 실행
```

### 전체 개념과의 연결

```
러닝 루프
├── 결정화 (§2)          ← 반복 패턴을 스킬로 굳힘
├── 자가 진화 스킬 (§8)  ← 기존 스킬을 피드백으로 수정
├── 3층 기억 저장 (§9)   ← 세션 내용을 적절한 레이어에 분류
└── 알림 UX (§5)         ← 결정화/진화 결과를 사용자에게 보고
```

## 11. FTS5 검색 + 주입

### 개념

> 사용자가 말하는 순간, 관련된 스킬/규칙을 DB에서 찾아서 Claude 프롬프트에 자동으로 끼워넣는 것.

### FTS5란?

SQLite에 내장된 전문 검색(Full-Text Search) 엔진. 별도 설치 불필요.

```
일반 검색: "pnpm" 이라는 단어가 정확히 있는가?
FTS5:      "pnpm", "패키지 매니저", "노드 버전" 등 관련 내용을 의미 근처까지 검색
```

### 왜 필요한가?

스킬 파일이 아무리 많아도 Claude가 알아서 읽지 않으면 소용없다.

```
기존 하네스: 사용자가 "/invoke harness-boundary-check" 직접 호출 → 그때만 읽음

FTS5 주입:  사용자가 "이 함수 리팩토링해줘" 라고만 해도
            → Hook이 "리팩토링" 키워드 감지
            → DB에서 관련 스킬 검색
            → harness-boundary-check 자동 주입
            → Claude가 경계 체크를 알아서 수행
```

### 주입 흐름

```
사용자 입력
    ↓
UserPromptSubmit Hook
    ↓
키워드 추출
    ↓
SQLite FTS5 검색
    ↓
관련 스킬/규칙 발견
    ↓
프롬프트 앞에 삽입:
"⚠️ 관련 규칙: [스킬 내용]"
    ↓
Claude가 해당 규칙을 인지한 채로 작업 시작
```

### ai-dev-setting 진입점

기존 `claude-userpromptsubmit-reminders.sh` Hook을 **확장**. 새 Hook을 만들지 않는다.

```
기존: active 계획 있으면 알림
확장: active 계획 알림 + FTS5 검색 + 관련 스킬 주입
```

### 3층 기억과의 연결

| 검색 대상 | 레이어 |
|---|---|
| 스킬 파일 | 3층 스킬 기억 |
| 프로젝트 규칙 | 2층 영구 기억 |
| 과거 유사 대화 | 1층 세션 기억 |

## 12. 자율 에이전트 (Agent View + claude --bg)

### 개념

> Claude Code의 Agent View와 `claude --bg` 를 활용해 백그라운드에서 독립적으로 작업하는 에이전트.
> 원본 헤르메스의 "완전 자율 에이전트"를 Claude Code 안에서 구현한 버전.

### 동작 방식

```
외부 cron (오전 9시)
    ↓
claude --bg "업무 시작: 프로젝트 현황 체크하고 서브에이전트 배분해줘"
    ↓
매니저 에이전트가 각 프로젝트에 서브에이전트 생성:
claude --bg "my-app 오늘 할 일 처리해줘"
claude --bg "rim-kanban 이슈 정리해줘"
    ↓
서브에이전트들이 각자 작업 (Agent View에서 모니터링)
    ↓
판단 필요한 순간 → Agent View에서 질문 대기
    ↓
외부 cron (오후 6시)
claude --bg "업무 종료: 오늘 진행상황 아카이브해줘"
```

### Stop Hook과의 연결

세션 종료 시 Stop Hook이 자동으로 백그라운드 회고 세션을 생성:

```bash
# claude-stop-retrospective.sh
python3 ~/.hermes/save_session.py     # 세션 SQLite 저장
```

**역할 분리:**
- Python: 데이터 준비 (SQLite 쿼리, 패턴 집계, 프롬프트 조립)
- Claude: 판단 (결정화 여부, 스킬 내용 작성)

### 현실적인 자율 범위

| 상황 | 동작 |
|---|---|
| 판단 불필요한 작업 | 혼자 끝까지 완주 |
| 판단 필요한 순간 | Agent View에서 질문 대기 |
| 완료 | 결과 보고 |

> "완전 자동"이 아닌 **"대부분 자동 + 필요할 때만 사람 개입"** — 현실적인 자율 에이전트의 완성형.

### 한계 및 해결 방향

| 한계 | 해결 |
|---|---|
| 에이전트 간 직접 대화 불가 | SQLite messages 테이블로 비동기 메시지 패싱 |
| Claude Code 자체 스케줄러 없음 | 외부 cron으로 `claude --bg` 트리거 |
| 프롬프트 품질에 따라 결과 편차 | 헤르메스 스킬 축적으로 점진적 개선 |

## 13. 에이전트 간 메시지 버스

### 개념

> 에이전트들이 직접 대화할 수 없는 한계를 SQLite를 공용 게시판으로 활용해 해결.

### messages 테이블

```sql
CREATE TABLE messages (
    id          INTEGER PRIMARY KEY,
    from_agent  TEXT,       -- 보낸 에이전트
    to_agent    TEXT,       -- 받을 에이전트
    content     TEXT,       -- 메시지 내용
    status      TEXT,       -- unread / read
    created_at  DATETIME
);
```

### 메시지 패싱 흐름

```
서브에이전트A 작업 완료
    ↓
messages 테이블 INSERT
"to: 매니저, content: my-app 완료"
    ↓
매니저 에이전트가 SELECT로 수신 확인
    ↓
claude --bg "B야, A 결과 받아서 이어서 해줘"
```

### 설계 원칙

- **실시간 대화 불필요** — 비동기 메시지 패싱으로 충분 (팀장-팀원 카톡 모델)
- **SQLite = 공용 게시판** — 모든 에이전트가 읽고 쓰는 공유 상태
- **매니저가 중재** — 서브에이전트끼리 직접 소통하지 않고 매니저 경유
