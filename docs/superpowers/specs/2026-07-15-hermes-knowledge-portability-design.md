# Hermes 지식 이식성 · 그물망 · 생애주기 설계

> 작성일: 2026-07-15
> 개정일: 2026-07-15 (적대적 리뷰 반영 — 시크릿 노출·그물망 소비자·gitignore·PII 메커니즘 등 9건)
> 목적: 프로젝트에서 축적된 지식(메모리·규칙·스킬·대화원본)을 저장소와 함께 버전관리해 여러 컴퓨터로 이식하고, 잠든 학습 사슬을 복구하며, 무한 증가를 생애주기 린트로 관리한다.

## 개요

한 컴퓨터에서 클로드코드가 "쓸수록 똑똑"해지는 이유는 두 갈래다.

1. **네이티브 메모리** — Claude Code 베이스 하네스가 `~/.claude/projects/<경로키>/memory/*.md`에 쓰는 사람이 읽는 지식(담당자·규약·피드백). 이 프로젝트(zeroday-frontend)에 18개 존재.
2. **Hermes 엔지니어링** — Hermes 하네스가 `.hermes/state.db`(SQLite)와 `.hermes/skills/*.md`에 쌓는 세션기록·결정화 규칙·스킬.

문제는 **①은 저장소 밖에 있어 이식 불가**하고, **②의 지식 원본 상당수도 로컬 바이너리 SQLite에 갇혀** 다른 컴퓨터로 넘어가지 않는다는 것이다. 이 설계는 네 파트(A/B/C/D)와 하나의 SQLite 처리 방침으로 이를 해결한다.

이 문서는 **설계**만 담는다. 구현·검증은 후속 계획에서 다룬다.

## 배경: 검증된 사실과 오진 정정

논의 중 데이터로 직접 검증한 사실(향후 오판 방지를 위해 기록):

- **네이티브 메모리는 Hermes가 아니라 Claude Code 베이스 하네스가 쓴다.** Hermes 코드는 `~/.claude/projects/.../memory/`를 읽거나 쓰지 않는다. (Claude Code changelog에 제품 기능으로 존재: `MEMORY.md` 인덱스, `CLAUDE_MEMORY_STORES`)
- **`harness_rules`는 잠들지 않았다.** 결정화는 프로젝트 `state.db`가 아니라 `~/.hermes/global.db`에 쓴다. global.db에 **157행이 살아있고 2026-07-15까지 기록**된다. 프로젝트 `state.db`의 `harness_rules` 0행은 "쓰는 코드가 프로젝트 스코프를 겨냥하지 않음"이므로 **정상**이다. (초기 오진 정정)
- **skill_injection은 진짜 고장이다.** `hermes-search.py`의 INSERT는 `source` 컬럼을 쓰는데(`scripts/hermes-search.py:402-406`), 기존 설치본 `state.db`의 `skill_injection`엔 `source` 컬럼이 없다(`[id, session_id, skill_path, injected_at, correlated]`). → `no such column: source`로 예외 발생 후 조용히 삼켜짐(`hermes-search.py:409-410`). 마이그레이션(`hermes-init.py:125-126`)은 (재)설치 때만 돌아, C1 이전 설치본은 주입이 전혀 기록되지 않는다.
- **helpful/noop 피드백은 하류가 굶은 것이다.** `hermes-correlate.py:112-124`가 helpful_count/noop_count를 증가시키고 Stop 훅(`assets/hooks/claude-stop-retrospective.sh:104`)에 배선돼 있으나, 입력인 `skill_injection` 행이 위 버그로 비어 실행이 무의미하다.

## 관통 원칙

이 설계 전체를 관통하는 두 규칙:

1. **SQLite = 재생성 가능한 로컬 색인 / git 텍스트 = 영속·이식 원본.**
   SQLite는 버리지 않되 "원본"이 아니라 "원본에서 다시 빌드하는 캐시"로 위상을 재정의한다.
2. **바깥으로 나가거나 되돌릴 수 없는 행동은 사람 게이트, 로컬에 머물고 복구 가능한 행동은 자동.**
   - Part C(그물망 승격): 지식이 프로젝트 밖으로 배포 + PII 위험 → 되돌리기 어려움 → **리뷰 게이트**
   - Part D(로컬 압축): 로컬에 머물고 git 히스토리로 복구 가능 → 저위험 → **자동**
3. **git에 커밋되는 모든 텍스트는 커밋 전에 시크릿·자격증명 스캔을 통과한다.**
   원본 대화(`.hermes/history/`)는 붙여넣은 토큰·비밀번호·타 PII를 포함할 수 있으므로, export 시 **정규식+엔트로피 기반 시크릿 스캔**으로 탐지·마스킹한다. 사내 비공개 저장소라도 git 히스토리는 영구적이므로 예외 없이 적용. (리뷰 H1)

## Part A. 네이티브 메모리(md) 저장소 버전관리

### 목표
`~/.claude/projects/<경로키>/memory/*.md`를 저장소 안으로 끌어와 git으로 버전관리하고 여러 컴퓨터로 이식한다.

### 저장 위치
- `<repo>/.claude/memory/` — **실제 디렉터리**(git 추적, 실제 .md 파일). git은 심링크를 보지 않는다.

### 연결 방식: 심링크 우선 + 훅복사 자동 폴백
`~/.claude/projects/<경로키>/memory/`를 `<repo>/.claude/memory/`로 심볼릭 링크하여 Claude Code가 저장소 안으로 투명하게 쓰게 한다. 심링크 불가 환경에서는 훅 복사로 자동 폴백한다.

### 5중 안전장치 (설치 실패·사이드이펙트 금지 요구사항 충족)
1. **능력 탐지 + 자동 폴백** — 설치 시 심링크 생성을 시험. 실패 시(윈도우 마운트·정책) SessionStart→home, Stop→repo 훅 복사로 폴백. **설치는 반드시 성공한다.**
2. **비파괴 마이그레이션** — 심링크 전 기존 18개 파일을 repo 타깃으로 이동/병합. 충돌 시 내용 비교 후 보존. 어떤 경우에도 파일 유실 없음.
3. **멱등 재실행** — 상태 감지 수렴: `올바른 심링크→무동작` / `잘못된 심링크→교정` / `실제 디렉터리→마이그레이션` / `없음→생성`.
4. **자가 치유 가드** — SessionStart 훅이 심링크 유효성 점검·복구. 하네스가 디렉터리를 재생성해 심링크를 날려도 다음 세션에 자동 복원.
5. **클린 언인스톨** — `uninstall.sh`가 심링크를 실제 디렉터리로 되돌리고 파일 복사 후 제거.

### 경로키 처리
메모리 폴더명은 절대경로를 인코딩한 키(`-home-jjackkun-PROJECT-zeroday-frontend`)다. 심링크는 기기 로컬이고 가리키는 대상은 pull된 repo이므로, 설치 스크립트가 각 컴퓨터의 실제 cwd에서 키를 계산해 올바른 위치에 심링크를 생성한다.

### 동시 편집 충돌 (리뷰 M2)
같은 프로젝트를 두 컴퓨터에서 만지면 각자 `MEMORY.md`·메모리 .md에 써서 git 충돌이 날 수 있다. 대처:
- 메모리 .md는 파일 하나=사실 하나라 **파일 단위 충돌이 드물다**(서로 다른 파일을 건드리면 자동 머지).
- `MEMORY.md` 인덱스는 **개별 .md에서 재생성 가능한 파생물**이므로, 충돌 시 인덱스는 **재생성으로 해소**한다(수동 머지 대상 아님).
- 권고: 동일 프로젝트를 두 기기에서 동시 작업하지 않는다. 필요 시 pull 후 작업.

### 가정·리스크 (리뷰 L1)
Part A는 Claude Code의 **문서화되지 않은 내부 메모리 경로**(`~/.claude/projects/<키>/memory/`)에 결합한다. 제품이 경로 스킴을 바꾸면 심링크가 무효화될 수 있다. 자가 치유 가드(안전장치 4)가 다음 SessionStart에 감지·복구하도록 하되, **경로 스킴 변경은 이 설계의 알려진 취약점**으로 명시한다.

### gitignore 갱신 (리뷰 H3)
`.claude/memory/`는 기본 추적 대상이지만, Hermes 쪽 export 경로(`.hermes/history/`·`.hermes/rules/`)는 설치본 `.gitignore`의 `.hermes/*` 규칙에 걸려 무시된다. `presets/workflow/hermes.conf`의 `GITIGNORE_ENTRIES`에 **`!.hermes/history/`·`!.hermes/history/**`·`!.hermes/rules/`·`!.hermes/rules/**` 예외를 추가**해야 한다. 누락 시 export 파일이 추적되지 않아 이식이 조용히 실패한다.

## Part B. 스킬 주입 → 유용도 피드백 사슬 복구

### 범위
- **B-1 `harness_rules`는 정상 작동 → 스코프 제외.** (단 "global.db에 갇힘"은 Part C가 다룸)
- **B-2 근본 버그 수리** + **B-3 자동 회복.**

### B-2: skill_injection 스키마 드리프트 수리
- **원인**: 코드는 `source` 컬럼에 INSERT하나 기존 DB엔 컬럼 부재 → 무음 실패.
- **해법**:
  1. **재설치 의존 제거** — 기존 DB의 스키마를 현재 버전으로 정합화. `skill_injection`은 **git 텍스트 원본이 없는 로컬 런타임 테이블**이므로(SQLite 처리 절 참조), "텍스트에서 재색인"이 아니라 **현재 스키마로 drop+recreate**한다(내용이 비어있거나 손상돼 안전). 이를 SessionStart의 스키마 점검에 포함해 재설치를 기다리지 않는다. (리뷰 M3 — 재색인과 런타임 테이블 재생성을 구분)
  2. **무음 실패 표면화** — `hermes-search.py:409` 근처의 예외 삼킴을 진단 로그로 노출(스키마 불일치를 조기 감지).

### B-3: 유용도 피드백 자동 회복
- `hermes-correlate.py`는 배선 완료 상태이므로, B-2 수리로 `skill_injection` 입력이 공급되면 helpful/noop 집계가 자동으로 살아난다.
- **후속 튜닝(구현 후 데이터 기반)**: correlate의 `MIN_KEYWORD_OVERLAP>=2`, assist 훅 트리거 조건(`--max 1`, `--once-per-session`, 세션 상한 3)이 실사용에 맞는지 데이터를 보고 조정. "기록이 흐른다 ≠ 유용하다"를 검증한다.

## Part C. 계층형 지식 그물망

### 목표
프로젝트에서 태어난 지식 중 **일반적인 것만** 전역 그물망으로 승격시켜, 프로젝트를 넘어 재사용한다. 이미 존재하는 `~/.hermes/global.db`의 `harness_rules` 157행이 그물망의 살아있는 씨앗이며, 현재 로컬 바이너리에 갇혀 있다.

### 소비자 확정: Hermes 주입 파이프 (리뷰 H2)
그물망의 소비자를 **Hermes로 확정**한다. 두 개의 별개 통로를 뒤섞지 않는다:
- **채택**: 승격된 일반 규칙 → **별도 그물망 git 저장소**의 텍스트(`rules/*.md`) → 각 컴퓨터가 pull → **각 기기의 `~/.hermes/global.db`로 재색인** → **Hermes 주입/회상이 소비**. (harness_rules는 Hermes가 먹는 데이터이므로 이 경로가 정합적)
- **비채택(혼동 주의)**: `CLAUDE_MEMORY_STORES`는 **Claude Code 네이티브 메모리**가 먹는 별개 통로다. Hermes의 harness_rules를 소비하지 않는다. 네이티브 메모리(Part A)를 팀·기기 간 공유하려는 별도 목적에는 쓸 수 있으나, **이 그물망(규칙)의 소비 통로는 아니며 본 설계 범위 밖**으로 둔다.

### 도달 범위: 내 여러 컴퓨터 (git 저장소)
- 위 그물망 git 저장소를 각 컴퓨터가 pull하여 로컬 `~/.hermes/global.db`에 재색인.
- **미래 확장(팀 전체)**: 같은 git 뼈대 위에 승인·머지·컴플라이언스 계층을 얹으면 됨. "내 여러 컴퓨터"는 "팀"의 부분집합이므로 지금 구조가 확장 경로를 막지 않는다. 처음부터 팀을 목표하면 거버넌스 무게에 눌리므로 2단계로 시작한다.

### 승격 방식: 리뷰 게이트 (결정화 지점에 부착)
`hermes-crystallize.py`가 이미 품질 게이트(SKIP→crystallized=-1)를 판정하는 그 지점에 승격 게이트를 한 겹 얹는다:

1. **🚨 1차 임무 = PII 차단(하드 제약).** 실존 인물·개인 식별 정보(예: "장정훈 차장")는 **어떤 경우에도 그물망 진입 금지.** 통과 조건이 아니라 **탈락 조건을 먼저** 본다.
   - **메커니즘 (리뷰 M1)**: 하드 제약을 불완전한 LLM 판단 하나에만 맡기지 않는다. **2단계**로 방어한다:
     (a) **값싼 사전 필터** — 한국어 직함/호칭 휴리스틱(`차장`·`부장`·`대리`·`님` 등)·이메일·전화·사번 정규식으로 명백한 PII 1차 차단.
     (b) **LLM 분류** — 통과분에 대해 "이 지식이 특정 개인·프로젝트에 종속되는가"를 판정.
   - **보수적 기본값**: 둘 중 하나라도 불확실하면 **승격 안 함(로컬 유지)**. 즉 블록리스트가 아니라 **"일반적이라고 확인된 것만 통과"하는 허용리스트** 방식.
2. **2차 = 일반성 판정.** PII를 통과한 것 중 프로젝트를 초월하는 일반 지식만 승격. `type: feedback`(일하는 방식)은 승격 후보, `type: project`·담당자는 프로젝트 로컬 유지. (판단 기준이 이미 메모리 `type:` 태그에 내장)

### 저장 형태
- 그물망 원본은 **별도 그물망 git 저장소**의 `rules/*.md` 텍스트로 export되어 이식된다(게이트 통과분만). **global.db의 157행 전부가 아니라 게이트를 통과한 일반 규칙만** 올라간다.
- **미통과 규칙**(PII·프로젝트한정)은 각 기기의 로컬 `~/.hermes/global.db`에 남으며 그물망으로 이식되지 않는다. 이는 의도된 경계다. (SQLite 처리 절 처분표 참조)

## Part D. 지식 생애주기 린트 (건강검진)

### 원칙
- **하드 삭제 금지.** 노화한 지식은 삭제가 아니라 **짧은 요약으로 압축**한다.
- 원문은 **git 히스토리에 영구 보존**된다: 작업 트리 = 요약본, git 히스토리 = 압축 전 원문. "버리면 안 된다"가 이중으로 보장된다.
- **전제 (리뷰 L2)**: 이 안전망은 해당 저장소 히스토리를 **절대 squash·prune·force-push하지 않음**을 전제한다. 히스토리를 재작성하면 압축 전 원문 복구가 깨지므로, 그물망/히스토리 저장소엔 **히스토리 재작성 금지**를 운영 규칙으로 못박는다.

### 대상
- **git 텍스트(`.hermes/history/`)가 린트 대상**이다. SQLite가 아니다.

### 압축 단위: 주제별
- 오래된 세션들을 주제로 클러스터링한 뒤 "이 주제로 배운 것 전체 → 짧은 요약 하나"로 압축.

### 실행: 자동 (겁 많은 자동)

## SQLite 처리 방침

### 대원칙
SQLite는 버리지도 git에 올리지도 않는다(gitignore 유지). "원본"에서 "재생성 가능한 로컬 색인"으로 강등한다.

### 테이블별 처분

| SQLite 테이블 | 처분 | 원본(source of truth) | 이식 |
|---|---|---|:---:|
| `session_history` (핑퐁 원본) | git 텍스트로 export → SQLite는 FTS5 색인 | `.hermes/history/*.jsonl` | ✅ |
| `harness_rules` (157행) | **게이트 통과분만** 그물망 repo로 export; 미통과분(PII·프로젝트한정)은 로컬 global.db 유지 | 그물망 git 저장소 `rules/*.md` | 통과분만 |
| `skill_index` | 그대로 색인 유지 | 이미 `.hermes/skills/*.md`(git) | ✅ |
| `session_summary` | 로컬 전용 (재생성 가능) | — (파생물) | ❌ |
| `pattern_count`/`pattern_session` | 로컬 전용 (진행 상태, 리셋 감내) | — | ❌ |
| `recall_marker`/FTS 내부 | 로컬 전용, 항상 재생성 | — | ❌ |
| `skill_injection` | 로컬 전용 (**correlation 입력 원장** — B-3의 핵심 입력, 단순 로그 아님) | — | ❌ |
| `dream_log` | 로컬 전용 (감사 로그) | — | ❌ |
| `loops`/`messages` | 미사용, 현행 유지 | — | ❌ |

지식 원본 3종(history·rules·skills)만 git 텍스트로 빠지고, 런타임 상태·색인·로그는 SQLite 로컬에 남는다.

### 쓰기 경로 (배치 export)
git은 메시지 단위 쓰기에 부적합하므로 기존 훅 지점에 export를 얹는다:
- **세션 중**: SQLite에 빠르게 누적 (현행 유지)
- **세션 끝(Stop 훅)**: 해당 세션 원본 턴을 **시크릿 스캔·마스킹(관통 원칙 3)을 거쳐** `.hermes/history/<날짜>-<session>.jsonl`로 export
- **결정화 시점**: `harness_rules` INSERT 옆에 `.hermes/rules/<key>.md` 텍스트 쓰기 추가
- 텍스트가 권위 있는 원본, SQLite는 검색용 색인.
- **git 추적 전제**: 위 export 경로는 `.gitignore`에서 `.hermes/*` 예외로 풀려 있어야 한다(Part A "gitignore 갱신", 리뷰 H3).

### 재색인 흐름 (이식 + 스키마 드리프트 동시 해결)
```text
새 컴퓨터:
  프로젝트 repo pull → .hermes/history·skills 텍스트 도착
    → 프로젝트 state.db 재색인 (session_history FTS5 + skill_index)
  그물망 repo pull → rules/*.md 도착
    → ~/.hermes/global.db 재색인 (harness_rules)
  (둘 다 hermes-init --reindex 또는 첫 SessionStart에서 현재 스키마로 빌드)
    → 즉시 "아는 상태"로 시작
```
두 저장소(프로젝트 repo = history·skills, 그물망 repo = rules)가 각각 대응하는 SQLite(state.db, global.db)로 재색인된다. SQLite가 재생성 가능한 색인이 되므로 `source` 컬럼 없는 낡은 DB도 현재 스키마로 리빌드하면 된다 — 깨지기 쉬운 in-place `ALTER`에 의존하지 않는다.

### 정합성 유보
- **부피**: `.hermes/history/`가 계속 커진다 → Part D 린트가 이 텍스트를 주제별로 압축.
- **이중 쓰기**: Stop에서 텍스트 export 실패 시 그 세션은 SQLite에만 남는다 → export를 멱등 재시도로 설계하고, 리빌드 시 "SQLite에만 있고 텍스트에 없는 세션"을 역-export하는 보정 패스를 둔다.
- **시크릿 스캔 불완전성 (리뷰 H1 잔여위험)**: 정규식+엔트로피 스캔은 완벽하지 않다. 신규 형식의 토큰을 놓칠 수 있으므로, (a) 스캔 룰을 갱신 가능하게 두고, (b) 스캔이 불확실하다고 표시한 항목은 **원본 대신 요약만 커밋**하는 보수적 폴백을 둔다. 사내 비공개 저장소라는 전제가 잔여위험의 완충이지만 무효화 근거는 아니다.
